# Gate risky Pi actions with an explicit Full Access mode

Standard Pi execution permits routine reads, searches, edits, and file creation inside the selected workspace, while shell commands and writes outside the workspace require an in-thread approval and credential-file access remains blocked. TIA credential storage remains inaccessible. An explicit Full Access switch skips approval gates for users who choose Codex-like unrestricted local execution without moving process or credential control into the renderer; the choice is stored per thread, new threads default to Standard Access, and no global switch broadens unrelated workspaces.

Command approval is owned by Electron main, not the renderer or Pi. The app stores structured Bash argv-prefix rules scoped to a workspace; executable user regex is not a supported rule format. Matching decisions use the strict order `hard block > deny > ask > allow`, so Full Access and remembered approvals cannot override credential hard blocks, and explicit deny rules also remain effective in Full Access.

The approval interaction offers Deny, Allow once, Allow for session, and Always allow in this workspace. Session approvals are memory-only and disappear when the live runtime closes or restarts. Workspace approvals are persisted, visible in Settings with their source, scope, rationale, and last-use time, and can be revoked. There is deliberately no default global approval.

TIA derives a proposed reusable rule only after parsing a provably simple command into argv segments. Simple `&&`, `||`, and `;` chains are evaluated segment by segment and the strictest result wins. Commands containing redirects, substitutions, variables, wildcards, heredocs, pipes, background execution, or shell control flow can be allowed once but do not receive an automatically generated reusable rule. Shells, interpreters, privilege wrappers, package installers, destructive commands, and arbitrary-command runners are also excluded from automatic rule derivation. This conservative model complements future OS sandboxing; it is not a substitute for process isolation.
